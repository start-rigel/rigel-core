# Rigel Architecture

## 当前系统定义

Rigel 当前不是一个复杂的硬件专家系统。
Rigel 当前是一个围绕价格清单和 AI 推荐工作的最小系统。

当前系统只围绕这条链路：

1. 后台管理页面维护型号词库，支持 Excel 导入导出
2. `rigel-jd-collector` 从京东联盟获取原始商品与价格
3. 原始商品进入 PostgreSQL
4. `rigel-build-engine` 将原始标题整理成型号级价格清单
5. `rigel-build-engine` 接收来自前台用户页面的参数
6. `rigel-build-engine` 组织 `用户需求 + 价格清单` 请求 AI API
7. `rigel-console` 向前台展示推荐结果，并向后台提供受保护的管理入口

## 服务边界

### `rigel-jd-collector`

负责：

- 消费已维护的型号词库
- 京东联盟/OpenAPI 调用
- 原始商品查询
- 原始价格快照入库

不负责：

- 价格清单整理
- AI 分析
- 页面展示

### `rigel-build-engine`

负责：

- 接收用户需求参数
- 整理原始硬件信息
- 形成型号级价格清单
- 构造 AI 输入
- 请求 AI API
- 返回结构化推荐结果

不负责：

- 直接抓取外部平台
- 承担前端界面

### `rigel-console`

负责：

- 前台推荐页面入口
- 后台管理页面入口
- 管理端登录态校验
- 接收前台用户输入
- 调用 `rigel-build-engine`
- 展示 AI 返回结果

不负责：

- 数据抓取
- 数据聚合
- AI 决策

## 当前最重要的数据产物

当前最重要的数据不是抓取页面，也不是后台列表。
当前最重要的数据产物是型号级价格清单，例如：

- `Ryzen 5 7500F -> 899`
- `RTX 4060 -> 2399`
- `B650M -> 699`
- `DDR5 6000 32G -> 499`

AI 消费的是这份清单，而不是原始商品明细。

## 前台与后台边界

当前页面体系必须拆成两类：

- 前台用户页面
- 后台管理页面

### 前台用户页面

前台面向普通用户，特点是：

- 不要求登录
- 只提供配置推荐相关能力
- 不暴露词库、采集、导入导出等管理功能

### 后台管理页面

后台面向项目维护者，特点是：

- 必须登录后才能访问
- 负责词库维护、导入导出、采集触发、后续运营配置
- 不与前台推荐页面放在同一个交互入口中

当前推荐站点与后台管理站点可以共用一个服务，但页面路由、鉴权和导航必须明确分离。

## 匿名使用与 AI 防滥用

Rigel 第一版默认面向不懂装机、也不想注册登录的普通用户。
因此当前产品原则是：

- 用户不需要登录即可直接使用推荐功能
- 不为了控费而强制增加注册门槛
- 不把 AI 接口直接裸露给前端无限调用

当前防滥用与控费方案采用四层结构：

1. 前端匿名会话层
2. `rigel-console` 接入限流层
3. `rigel-build-engine` AI 调用判定与缓存层
4. CDN / WAF 风险拦截层

### 1. 匿名会话

首次访问时，前端与 `rigel-console` 应为用户生成匿名会话标识，例如：

- `anonymous_id`
- `session_id`

该标识用于：

- 识别同一匿名用户的短期行为
- 辅助限流
- 命中相同请求缓存

当前不要求用户注册，也不要求手机号或邮箱验证。

### 2. 限流维度

对高成本接口，不能只看单一 IP。
当前至少采用以下三维联合限流：

- IP
- `anonymous_id`
- 设备指纹或浏览器特征摘要

原因：

- 仅按 IP 会误伤共享网络用户
- 仅按 Cookie 很容易被脚本清空
- 联合维度更适合匿名产品

### 3. AI 请求分层

当前要把请求分成两层：

- 低成本层：
  - 页面访问
  - 表单修改
  - 价格清单查询
  - 已有结果缓存命中
- 高成本层：
  - 触发新的 AI 推荐生成

低成本层只做轻量限流。
高成本层必须做：

- 请求归一化
- 缓存查询
- 短时间重复请求去重
- 匿名配额判断
- 风险升级挑战

### 4. 缓存优先

对 `预算 + 用途 + 装机模式 + 品牌偏好 + 特殊需求 + 备注` 这类输入，
后端应先做归一化，然后计算缓存键。

命中缓存时：

- 直接返回已有推荐结果
- 不重新请求 AI

这条规则是匿名模式下控费的核心，不允许绕过。

### 5. 软额度与冷却

匿名用户默认可以直接使用，但高成本 AI 生成必须有软额度，例如：

- 单匿名会话每小时最多若干次新的 AI 生成
- 单 IP 每分钟最多若干次高成本请求

超过阈值时：

- 优先进入短冷却
- 返回友好提示
- 不立即要求登录

当前不建议第一版直接使用硬注册墙。

### 6. 风险升级

只有在明显异常时，才升级挑战手段，例如：

- 并发异常高
- 指纹缺失
- 短时间大量切换参数
- 单 IP / 单设备命中高风险阈值

升级动作可以包括：

- 验证码
- 人机挑战页
- WAF 封禁

正常用户不应默认看到这些步骤。

### 7. 服务职责

`rigel-console` 负责：

- 匿名会话签发
- 前端请求节流
- 初步限流
- 风险挑战入口
- 后台登录态校验
- 后台管理路由保护

`rigel-build-engine` 负责：

- 请求归一化
- 缓存命中判断
- AI 调用幂等与去重
- AI 调用配额控制
- 结果缓存

`Redis` 负责：

- 限流计数
- 幂等锁
- 匿名配额
- 短期缓存

### 8. 降级策略

当 AI 成本或风险异常升高时，系统允许降级，但不能直接不可用。

当前允许的降级方式：

- 优先返回缓存结果
- 返回基础模板化推荐草案
- 延长匿名用户冷却时间
- 暂时关闭高风险请求入口

第一版不要求完全实时，但要求对正常用户可持续可用。

## 当前正式使用的京东联盟接口

- `jd.union.open.goods.query`
- `jd.union.open.goods.promotiongoodsinfo.query`
- `jd.union.open.category.goods.get`

未来返佣链接预留：

- `jd.union.open.promotion.common.get`

## 当前明确不做

当前架构范围内，不做：

- 闲鱼接入
- 浏览器抓取京东
- 独立 AI 服务
- 重型兼容规则引擎
- 多数据源联合推荐
- 复杂后台系统
