# 电脑配置平台项目方案文档

## 1. 项目概述

### 1.1 项目名称

中国电脑配置智能推荐平台（暂定）

### 1.2 项目愿景

打造一个面向中国用户的电脑配置智能平台，聚合京东新品价格与闲鱼二手行情，结合配件兼容性规则、性能分级模型和 AI 生成能力，为用户输出可执行、可解释、可替代的装机方案。

### 1.3 核心目标

本项目不是单纯的比价网站，也不是一个普通的 AI 聊天工具，而是一个具备以下能力的系统：

1. 自动采集配件价格与行情。
2. 建立标准化配件数据库。
3. 管理配件兼容关系与约束规则。
4. 根据预算、用途、偏好自动生成装机方案。
5. 允许 AI 基于数据库结果输出可读性强的推荐理由与替代方案。
6. 支持全新方案、二手方案、混搭方案三类输出。

### 1.4 解决的问题

当前用户在装机时通常会遇到以下痛点：

- 不知道同预算应该怎么配。
- 不懂 CPU、主板、显卡之间的兼容关系。
- 不知道闲鱼二手值不值得买。
- 不知道某个预算应该优先堆显卡还是 CPU。
- 不知道不同用途的配置思路差异。
- 不知道哪些型号是套路机、矿卡风险、洋垃圾平台。

本项目要解决的是：

- 让用户输入预算和需求后，系统自动生成可执行配置。
- 让配置结果具备价格依据和规则依据。
- 让 AI 不是“乱推荐”，而是读取数据库后在约束范围内推荐。

## 2. 产品定位

### 2.1 产品本质

这是一个“配件数据库 + 规则引擎 + AI 推荐层”的组合系统。

### 2.2 与现有平台的区别

传统装机网站通常只能做：

- 静态装机单
- 人工编辑推荐
- 简单兼容校验
- 无实时价格
- 无二手行情
- 无 AI 解释

本项目的差异化：

1. 价格是动态的，不是手工维护。
2. 支持京东新品与闲鱼二手的联合推荐。
3. 支持基于场景的自动组合，而不是固定模板。
4. AI 输出建立在真实数据库和规则之上。
5. 支持替代方案与价格波动后的自动重组。

### 2.3 用户画像

#### 用户类型 A：小白用户

- 预算明确，但不懂配件。
- 希望输入“4000 元打游戏”直接得到配置。
- 需要简单解释为什么这么配。

#### 用户类型 B：进阶用户

- 对配件有基本认知。
- 希望对比全新和二手方案。
- 希望手动锁定某个配件后继续自动补全。

#### 用户类型 C：电脑维修 / 装机服务从业者

- 需要快速生成可发给客户的装机单。
- 希望按预算快速出多套方案。
- 希望拿到替代配件与行情参考。

## 3. MVP 范围定义

### 3.1 第一阶段只做什么

建议第一阶段只做以下范围：

1. 只支持台式机。
2. 只支持常见用途：
   - 打游戏
   - 办公
   - 设计剪辑
3. 只支持主流配件：
   - CPU
   - 主板
   - 显卡
   - 内存
   - SSD
   - 电源
   - 机箱
   - 散热器
4. 数据源优先京东。
5. 闲鱼仅作为参考价，不直接参与自动下单级推荐。
6. AI 仅负责生成解释，不直接负责兼容计算。

### 3.2 第一阶段不做什么

- 不做笔记本推荐。
- 不做显示器、键鼠、耳机等外设。
- 不做实时下单跳转的复杂联盟功能。
- 不做多平台混合结算。
- 不做自动识别所有非标二手标题。
- 不做复杂用户社区。

### 3.3 第二阶段扩展

- 加入闲鱼标准化二手商品。
- 加入二手风险评分。
- 加入本地面交支持。
- 加入用户锁定配件后二次生成。
- 加入历史价格趋势图。
- 加入配置收藏、分享、导出图片。

### 3.4 第三阶段扩展

- 加入多平台比价。
- 加入配置利润空间分析。
- 加入装机商专属后台。
- 加入自动报价单生成。
- 加入地域偏好、商家评级、售后等级。

## 4. 核心功能模块

项目整体拆分为 5 个模块，分别负责不同职责，既方便并行开发，也方便后续独立扩展。

### 4.0 模块命名规范

为了方便后续代码仓库管理、服务拆分、Docker 部署和文档维护，所有模块统一增加项目代号前缀：`rigel-`。

#### 模块命名总表

| 中文名称 | 英文名称 | repo / 服务名 | 说明 |
|---|---|---|---|
| Rigel-闲鱼采集模块 | Goofish Collector | `rigel-goofish-collector` | 负责闲鱼二手配件行情采集 |
| Rigel-京东采集模块 | JD Collector | `rigel-jd-collector` | 负责京东新品配件价格采集 |
| Rigel-配置引擎模块 | Build Engine | `rigel-build-engine` | 负责标准化、规则校验和配置生成 |
| Rigel-AI顾问模块 | AI Advisor | `rigel-ai-advisor` | 负责配置说明、建议生成和问答 |
| Rigel-应用模块 | Console | `rigel-console` | 负责前台、后台和 API 服务 |

#### 命名原则

1. 中文名用于方案文档、产品说明、团队沟通。
2. 英文名用于模块定位和架构层命名。
3. `repo / 服务名` 用于 Git 仓库、Docker 服务、部署服务名、环境变量命名。
4. 所有服务统一采用 kebab-case 命名风格，便于部署管理。
5. 若未来平台正式品牌名确定，可保留模块后缀结构，仅替换前缀 `rigel-`。

#### 推荐使用方式

- 文档中优先写中文模块名，例如：Rigel-配置引擎模块。
- 架构图中可写英文模块名，例如：Build Engine。
- 代码仓库与部署中统一使用 repo / 服务名，例如：`rigel-build-engine`。


### 4.1 闲鱼数据采集模块

负责抓取闲鱼二手商品与行情数据。

该模块建议基于现有开源项目 **ai-goofish-monitor** 进行 fork 和二次改造，而不是从零开发。二次改造目标不是做通用闲鱼监控器，而是服务于“电脑配件行情采集”这一明确场景。

#### 模块职责

- 基于关键词采集闲鱼商品列表
- 获取商品标题、价格、链接、图片、卖家位置等基础信息
- 对热门型号定时刷新
- 过滤求购、换物、整机混卖、坏件等无效信息
- 输出用于后续标准化处理的原始商品数据
- 为系统提供二手价格区间参考

#### 改造重点

- 从通用监控改为“配件型号驱动采集”
- 增加 CPU / 显卡 / 主板 / 内存 / SSD 等型号关键词体系
- 增加无效标题过滤规则
- 增加价格区间统计与中位数聚合
- 增加与平台数据库的对接能力
- 增加任务调度和失败重试能力

#### 第一阶段定位

第一阶段只做参考行情层，不直接参与自动配置主链路，不做自动下单级推荐。

### 4.2 京东数据采集模块

负责抓取京东新品商品、价格和基础属性信息。

该模块是第一阶段的核心数据来源，优先保证主流型号的价格可用性和刷新稳定性。

#### 模块职责

- 根据标准型号关键词检索京东商品
- 获取商品标题、SKU、价格、店铺、链接、图片等信息
- 定时刷新热门型号价格
- 识别自营 / 非自营 / 旗舰店等信息
- 输出原始商品数据与价格快照
- 为配置生成提供新品价格基准

#### 第一阶段目标

- 优先覆盖 CPU、主板、显卡、内存、SSD、电源、机箱、散热
- 优先覆盖 50-100 个热门标准型号
- 保证主流配件价格刷新链路可跑通

### 4.3 数据与推荐引擎模块

这是整个平台的核心模块，负责把采集来的原始商品数据转成可用配件数据，并最终生成配置结果。

这个模块可以理解为“数据处理层 + 规则决策层”的组合。

#### 模块职责

- 商品标题解析
- 品牌、型号、属性识别
- 商品去重与标准化映射
- 标准配件库维护
- 行情聚合
- 兼容规则校验
- 预算分配
- 风险过滤
- 候选配件筛选
- 整机组合与推荐排序
- 主方案、稳定方案、性价比方案输出

#### 模块边界

- 兼容判断必须由该模块负责
- 风险标签命中必须由该模块负责
- AI 不参与该模块中的兼容计算和主决策

### 4.4 AI 模块

负责将结构化配置结果转化为用户可读内容。

该模块是解释层，不是规则层，也不是配置主决策层。

#### 模块职责

- 输出配置说明
- 输出推荐理由
- 输出风险提示
- 输出升级建议
- 回答用户对当前配置的追问
- 生成更适合分享或展示的文案

#### 模块边界

- 不做配件搜索
- 不做兼容计算
- 不虚构型号和价格
- 不覆盖规则引擎结论

### 4.5 应用模块

负责把前面 4 个模块整合成可用的平台产品，包括前台、后台和 API 服务。

#### 模块职责

- 用户前台页面
- 管理后台页面
- 配置生成接口
- 配件查询接口
- 价格查询接口
- 任务状态查看
- 商品审核与映射纠错
- 登录、权限、日志、基础运营能力

#### 包含页面

##### 用户前台
- 首页
- 配置结果页
- 配件详情页
- 条件筛选页

##### 管理后台
- 数据总览
- 商品管理
- 配件库管理
- 规则管理
- 配置结果审查
- 任务监控页

## 7. 配件标准化策略

这是整个项目最关键的部分之一。

### 7.1 标准化目标

把杂乱的商品标题解析为统一型号。

### 7.2 标准化难点

- 同型号不同写法。
- 中英文混写。
- 盒装、散片、板 U 套装混写。
- 闲鱼标题大量省略、错别字、口语化。
- 商品标题同时出现多个型号。

### 7.3 建议方案

#### 第一步：规则字典匹配

维护品牌、系列、型号关键词字典。

#### 第二步：正则解析

识别：

- CPU 型号
- 显卡型号
- 主板芯片组
- 内存容量与代数
- SSD 容量与接口

#### 第三步：模糊匹配

对低置信度结果做模糊检索。

#### 第四步：人工审核纠错

后台允许人工修正映射结果。

#### 第五步：形成别名库

每次人工修正后沉淀到别名库，减少后续错误。

### 7.4 标准化置信度分级

- 0.9 以上：自动通过
- 0.7-0.9：待复核
- 0.7 以下：不参与自动配置

## 8. 兼容规则引擎设计

### 8.1 为什么规则引擎不能交给 AI

AI 可以解释配置，但不适合做底层兼容判定。原因：

- 规则需要确定性。
- 错误代价高。
- 相同输入必须得到相同结果。

因此：

- 兼容判断必须由规则引擎完成。
- AI 只基于结果解释，不自行决定底层兼容关系。

### 8.2 基础兼容规则

#### CPU ↔ 主板

- 插槽一致
- 支持代际一致
- BIOS 支持检查

#### 主板 ↔ 内存

- DDR4 / DDR5 匹配
- 最大容量限制
- 插槽数量

#### 显卡 ↔ 电源

- 建议功耗冗余
- 供电接口检查

#### 主板 ↔ 机箱

- M-ATX / ATX / ITX 匹配

#### 散热 ↔ 机箱

- 限高检查

#### 显卡 ↔ 机箱

- 长度检查

### 8.3 场景导向规则

不同用途的预算分配策略：

#### 游戏主机

- 显卡优先
- CPU 次优先
- 内存 16G/32G 合理即可

#### 办公主机

- 稳定优先
- 核显或低功耗优先
- SSD 优先于高端显卡

#### 剪辑设计

- CPU 与内存优先
- 显卡次优先
- SSD 容量优先级提高

### 8.4 风险过滤规则

自动过滤：

- 明确高风险型号
- 高故障率杂牌电源
- 疑似矿卡关键词
- 老平台高溢价方案

## 9. 配置生成算法

### 9.1 输入参数

- 预算
- 用途
- 是否接受二手
- 品牌偏好
- 锁定配件
- 城市
- 是否优先性价比
- 是否优先稳定

### 9.2 输出内容

- 配件清单
- 总价
- 预算占比
- 性能评分
- 替代方案
- 风险说明
- 升级建议

### 9.3 推荐流程

#### 第一步：确定配置模板

根据用途和预算确定基础策略。

#### 第二步：分配预算桶

例如游戏主机：

- 显卡 35%-45%
- CPU 15%-20%
- 主板 8%-12%
- 内存 8%-10%
- SSD 8%-10%
- 电源 8%-10%
- 机箱 + 散热 5%-8%

#### 第三步：筛选候选配件

根据预算桶筛选候选列表。

#### 第四步：兼容筛选

用规则引擎过滤不兼容组合。

#### 第五步：评分排序

评分维度：

- 性能分
- 价格分
- 风险分
- 品牌分
- 新旧状态分
- 未来升级空间分

#### 第六步：输出最佳方案与备选方案

至少输出：

- 主推荐
- 稳定方案
- 性价比方案

### 9.4 示例评分公式

综合分 = 性能权重 × 0.4 + 性价比权重 × 0.25 + 稳定性权重 × 0.15 + 风险修正 × 0.1 + 升级空间 × 0.1

## 10. AI 模块设计

### 10.1 AI 的职责

AI 不做底层配件搜索，不做兼容计算，AI 主要负责：

- 解释推荐理由
- 输出更自然的文案
- 回答用户追问
- 生成备选方案说明

### 10.2 AI 输入内容

结构化 JSON：

- 用户需求
- 推荐配置结果
- 各部件价格
- 各部件核心参数
- 替代配件列表
- 风险标签

### 10.3 AI 输出模板

AI 应输出：

1. 简要结论
2. 配置清单
3. 推荐理由
4. 适合的用途
5. 升级建议
6. 风险提示

### 10.4 提示词原则

- 只能基于提供的数据回答。
- 不得虚构不存在的配件。
- 不得推荐规则引擎判定为不兼容的组合。
- 对二手配件必须明确风险提示。
- 对预算不足必须明确指出取舍。

### 10.5 示例提示词骨架

你是一个电脑装机顾问。你只能根据系统传入的结构化配置结果进行解释，不得虚构型号，不得更改兼容关系。请用清晰、口语化、专业但不生硬的方式，向用户说明这套配置为什么这样搭配，并给出 1 套备选建议。

## 13. 数据采集策略

### 13.1 京东采集建议

优先使用关键词搜索获取候选商品，再通过价格接口或详情页补全价格和属性。

采集策略：

- 定时抓热门型号
- 用户搜索触发增量抓取

建议刷新频率：

- 热门显卡：15-30 分钟
- CPU / 主板：30-60 分钟
- 机箱 / 散热：6-12 小时

### 13.2 闲鱼采集建议

第一阶段只做参考行情，不直接进入自动配置主链路。

采集策略：

- 针对热门型号采集前 N 条商品
- 标题解析型号
- 去除明显无效贴
- 形成价格区间而非单一价格

### 13.3 反爬与稳定性策略

- 控制请求频率
- 加重试与退避
- 分类任务队列
- 失败记录与报警
- 不依赖单一页面解析逻辑

## 14. 风险控制

### 14.1 法务与平台风险

- 数据抓取应尽量基于公开可访问数据。
- 对接官方接口优先于页面抓取。
- 对高风险来源使用降级方案。
- 系统中应保留来源链接和抓取时间。

### 14.2 业务风险

- 标题解析错误导致错配。
- 二手商品虚假描述。
- 价格波动导致配置失真。
- AI 解释与规则结果不一致。

### 14.3 技术风险

- 采集失效
- 映射准确率不够
- 规则复杂度过高
- 搜索性能问题

### 14.4 风险应对措施

- 引入人工审核闭环
- 为二手商品增加低置信度隔离区
- 保留原始商品文本与映射记录
- 为每次配置结果保留快照，保证可追溯

## 19. 结论

这个项目的核心不是“AI 会不会推荐配置”，而是：

- 能否建立可靠的配件标准库；
- 能否把非结构化商品数据映射成标准型号；
- 能否用规则引擎稳定生成兼容配置；
- 能否让 AI 只在正确的结果上做表达增强。

因此项目实施顺序必须是：

数据标准化 > 规则引擎 > 配置生成 > AI 表达层 > 二手深化。

如果直接从“让 AI 自动配电脑”开始做，大概率会失控；如果先把数据和规则打稳，AI 就会成为非常自然的增强层。

## 22. 兼容规则引擎设计文档

本章节把第 8 章进一步展开到可编码的程度。

### 22.1 规则引擎目标

规则引擎负责解决三个问题：

1. 这个组合能不能装。
2. 这个组合值不值得推荐。
3. 这个组合是否存在明确风险。

### 22.2 规则分层

#### 第一层：硬兼容规则

只要不满足，就直接淘汰。

示例：

- CPU 插槽与主板插槽不一致。
- 主板只支持 DDR4，但内存是 DDR5。
- 显卡长度超过机箱上限。
- 电源额定功率小于整机建议值。

#### 第二层：软兼容规则

不一定淘汰，但会扣分。

示例：

- B760 + i9 属于可以用，但不够均衡。
- 高端显卡搭配低质量电源。
- 高发热 CPU 搭配入门风冷。

#### 第三层：场景适配规则

根据用户用途调整优先级。

示例：

- 游戏优先显卡。
- 剪辑优先 CPU + 内存 + SSD。
- 办公优先稳定、低噪音、低功耗。

#### 第四层：风险规则

用于警告、降权或屏蔽。

示例：

- 洋垃圾平台。
- 闲鱼矿卡高风险。
- 杂牌电源。
- 明显溢价型号。

### 22.3 引擎执行顺序

建议按以下顺序执行：

1. 输入标准化
2. 候选池生成
3. 硬兼容过滤
4. 软兼容打分
5. 场景适配加权
6. 风险惩罚
7. 输出排序

### 22.4 核心兼容规则清单

#### CPU ↔ 主板

- socket 必须一致。
- 芯片组必须支持该代 CPU。
- 若存在 BIOS 风险标签，则默认降权或提示。

#### 主板 ↔ 内存

- memory\_type 必须一致。
- 内存总容量不得超过主板 max\_memory。
- 对于办公 / 游戏场景，内存频率过高但主板不支持 XMP 时降权。

#### CPU ↔ 散热

- 散热器扣具必须支持对应 socket。
- 当 CPU tdp 高于散热器推荐阈值时，降权或淘汰。

#### 显卡 ↔ 机箱

- gpu\_length\_mm <= max\_gpu\_length。
- slot\_thickness 不得超过可支持厚度。

#### 散热 ↔ 机箱

- 风冷高度 <= 机箱散热限高。
- 水冷排规格必须在机箱支持列表中。

#### 电源 ↔ 显卡 / 全机功耗

- 电源额定功率 >= 估算功耗 × 安全系数。
- 显卡供电接口必须可满足。
- 对高端显卡默认预留 20%-30% 冗余。

#### 主板 ↔ 机箱

- 板型必须兼容。

#### SSD ↔ 主板

- M.2 / SATA 接口必须存在。
- 对高性能 NVMe 场景，可用但跑不满速时降权提示。

### 22.5 性能与均衡规则

#### 游戏场景建议

- GPU 预算占比优先。
- CPU 不应明显拖累显卡。
- 16G 为最低建议，32G 为中高预算建议。

#### 剪辑设计建议

- CPU 与内存权重提高。
- 大容量 SSD 必须加分。
- 显卡不必绝对拉满。

#### 办公场景建议

- 优先稳定和低功耗。
- 独显不是必须。
- 杂牌高功耗平台应降权。

### 22.6 风险规则建议清单

#### 直接屏蔽 blocked

- 明确假标电源
- 无法识别型号的关键核心件
- 明确不兼容组合

#### 降权 downgrade

- X99 / 老服务器平台用于主流游戏预算配置
- 高疑似矿卡关键词
- 杂牌主板
- 非主流保修状态

#### 仅提示 warn

- 二手显卡来源不明
- 散片 CPU 无保
- 价格波动大

### 22.7 估算功耗逻辑

建议整机功耗估算公式：

整机估算功耗 = CPU 峰值功耗 + GPU 峰值功耗 + 主板 / 内存 / SSD / 风扇附加值

经验附加值建议：

- 主板 + 内存 + SSD：50-80W
- 多风扇机箱：10-20W
- 水冷：10W 左右

电源建议值：

- 推荐额定功率 >= 整机估算功耗 × 1.3

### 22.8 推荐排序逻辑

每套候选配置都应生成以下分数：

- compatibility\_score
- performance\_score
- value\_score
- stability\_score
- upgrade\_score
- risk\_penalty

最终分数示例：

final\_score = compatibility\_score × 0.25 + performance\_score × 0.25 + value\_score × 0.2 + stability\_score × 0.15 + upgrade\_score × 0.15 - risk\_penalty

### 22.9 规则引擎实现建议

#### 实现方式

推荐采用“代码规则 + 配置表”混合实现。

原因：

- 纯数据库表达复杂规则会很难维护。
- 纯代码写死又不利于运营后台调整。

建议：

- 硬兼容规则写在代码层。
- 风险标签与权重写在配置表。
- 场景权重写在 scoring\_profiles 表。

### 22.10 规则引擎返回协议

每次运行后应返回：

- is\_compatible
- fail\_reasons[]
- warnings[]
- score\_breakdown{}
- applied\_rules[]

这样便于：

- 前端展示
- AI 解释
- 后台排查

## 23. 京东 / 闲鱼采集方案文档

本章节把采集模块拆成更可执行的策略。

### 23.1 采集原则

1. 优先官方接口，其次公开页面。
2. 优先采集热门标准型号，不追求全量全站。
3. 价格抓取要服务于配置推荐，而不是做平台级电商搜索引擎。
4. 保留抓取来源、时间和原始数据。

### 23.2 京东采集方案

#### 第一阶段目标

实现基于型号关键词的候选商品搜索与价格抓取。

#### 入口策略

- 从标准配件表生成关键词。
- 针对每个 part 维护一组搜索词。
- 搜索结果取前 N 条候选商品。

#### 搜索关键词示例

对于 RTX 4060：

- RTX 4060
- 4060 显卡
- RTX4060 8G
- 七彩虹 RTX 4060

对于 i5-12400F：

- i5-12400F
- 12400F CPU
- Intel 12400F 盒装

#### 京东抓取字段优先级

P0：

- title
- sku
- price
- shop\_type
- shop\_name
- url
- image

P1：

- promo\_text
- stock\_status
- original\_price
- category

P2：

- 评论数
- 自营标签
- 配送时效

#### 候选商品筛选逻辑

- 标题命中标准型号关键词。
- 排除明显套装、非单件商品。
- 优先自营 / 旗舰店。
- 标题出现错代型号则过滤。

#### 京东价格刷新频率建议

- 显卡：15-30 分钟
- CPU：30 分钟
- 主板：60 分钟
- 内存 / SSD：2 小时
- 电源 / 机箱 / 散热：6 小时

### 23.3 闲鱼采集方案

#### 第一阶段定位

只做参考行情层，不直接作为自动配置主链路。

#### 采集目的

- 给系统提供二手价格区间。
- 给用户展示“这件新品和二手行情差多少”。
- 给后续二手混搭配置做准备。

#### 闲鱼候选采集逻辑

- 针对热门型号抓前 N 条。
- 清洗广告、求购、换物、整机混卖。
- 标题必须包含核心型号或别名。
- 优先同城或高质量卖家结果。

#### 闲鱼价格处理原则

不要直接用最低价。

建议计算：

- 中位数价格
- 25% 分位价格
- 75% 分位价格
- 样本数

#### 典型无效标题过滤

- 求收
- 求购
- 置换
- 换物
- 回收
- 仅机箱
- 仅包装
- 坏件
- 不包好坏

### 23.4 采集任务设计

#### 任务类型

- keyword\_search\_job：关键词搜索任务
- product\_detail\_job：商品详情补全任务
- price\_refresh\_job：价格刷新任务
- mapping\_recheck\_job：映射重跑任务

#### 任务优先级

- P0：用户请求触发的实时采集
- P1：热门型号定时刷新
- P2：冷门型号低频刷新

### 23.5 搜索触发模式

#### 主动定时模式

每天定时刷新热点型号。

#### 被动触发模式

用户请求一个预算方案时，如果候选型号价格过旧，触发增量刷新。

#### 缓存策略

- 30 分钟内有有效价格快照，则不重复抓取。
- 价格过期时优先异步刷新，同时使用最近快照做降级。

### 23.6 商品去重与归并策略

去重维度：

- source\_platform + source\_product\_id
- URL 标准化
- 标题归一 + 店铺 + 图片近似

同型号多商品归并时建议：

- 京东优先低价且高信誉店铺
- 闲鱼优先样本中位价，不选极端低价

### 23.7 采集失败处理

- 超时：重试 2-3 次
- 页面结构异常：进入告警队列
- 价格为空：标记无效快照
- 触发风控：暂停该源任务并降频

### 23.8 合规与技术策略

- 尽量通过公开数据或授权接口获取。
- 控制频率，避免高并发冲击。
- 系统保留来源链接与更新时间。
- 不把采集系统设计成泛搜索引擎，而是限定在装机场景相关配件型号范围内。

## 24. AI 推荐提示词与输出协议文档

本章节定义 AI 在系统中的边界、输入格式和输出格式。

### 24.1 AI 的角色定位

AI 是解释器，不是判官。

它只能：

- 解释为什么推荐这套。
- 说明适合什么用途。
- 给出替代建议。
- 提醒风险和升级方向。

它不能：

- 自行新增数据库中不存在的配件。
- 修改规则引擎的兼容结论。
- 忽略风险标签。
- 自己编造价格。

### 24.2 AI 输入结构建议

建议后端整理统一 JSON 给 AI。

#### 示例结构

```json
{
  "user_request": {
    "budget": 5000,
    "use_case": "gaming",
    "build_mode": "new_only",
    "preferences": ["stable", "value"],
    "allow_used_gpu": false
  },
  "build_result": {
    "build_name": "5000元游戏主机",
    "total_price": 4988,
    "estimated_power": 380,
    "scores": {
      "performance": 82,
      "value": 86,
      "stability": 79,
      "risk": 18
    },
    "items": [
      {
        "category": "CPU",
        "name": "Ryzen 5 5600",
        "price": 699,
        "reason": "游戏性能均衡，预算利用率高"
      },
      {
        "category": "GPU",
        "name": "RTX 4060",
        "price": 1999,
        "reason": "当前预算内综合游戏表现较优"
      }
    ],
    "warnings": [
      "该方案后续升级高端显卡时建议同步升级电源"
    ],
    "alternatives": [
      {
        "name": "RX 6750 GRE",
        "reason": "部分分辨率下性价比更高"
      }
    ]
  }
}
```

### 24.3 系统提示词建议

```text
你是一个电脑装机推荐助手。你必须严格根据系统传入的结构化数据进行说明，不得虚构配件、价格和兼容关系。你不能推荐未出现在输入数据中的部件，不能忽略 warnings 和风险提示。你需要用清晰、口语化、专业但不生硬的中文输出结果，帮助用户理解这套配置为什么这样搭配。
```

### 24.4 输出格式建议

AI 输出统一为以下结构：

```json
{
  "summary": "这是一套偏游戏优先的 5000 元主机，预算主要堆在显卡上，适合 1080P 到 2K 主流游戏。",
  "reasons": [
    "这套配置优先保证显卡预算，所以游戏表现更稳。",
    "CPU 选择的是性价比较高的中端型号，不会明显拖累显卡。",
    "电源和散热控制在够用范围，整体更均衡。"
  ],
  "fit_for": [
    "1080P 主流游戏",
    "日常办公",
    "轻度剪辑"
  ],
  "risks": [
    "后续如果升级更高功耗显卡，建议同步升级电源。"
  ],
  "upgrade_advice": [
    "后续可优先把内存升级到 32G。"
  ],
  "alternative_note": "如果更在意纯性价比，也可以考虑替换为另一张同价位显卡。"
}
```

### 24.5 AI 输出风格要求

- 语言口语化，但不要太随意。
- 不要写成论坛吵架风格。
- 不要输出太多空话。
- 每个推荐理由必须对应结构化数据中的事实。
- 二手方案必须显式提示风险。

### 24.6 AI 失败降级策略

若 AI 服务不可用，系统仍应返回：

- 结构化配置表
- 规则引擎警告
- 模板化说明文案

模板化说明可以由后端拼接：

- 该方案总价 xxx 元
- 主要偏向 xxx 场景
- 预算重点投入在 xxx
- 当前风险提示为 xxx

### 24.7 AI 与规则引擎协作边界

推荐流程应为：

- 规则引擎先出结构化方案
- AI 再做解释与表达优化

绝不能反过来让 AI 先猜方案，再交给规则引擎修正。

### 24.8 AI 版本管理建议

每次调用 AI 都记录：

- prompt\_version
- model\_name
- request\_payload\_hash
- output\_text
- created\_at

这样便于：

- 回溯错误输出
- 对比不同提示词版本效果
- 审核线上生成质量

## 25. 直接进入开发前的实施建议

### 25.1 第一个开发 Sprint 建议

建议第一个 Sprint 只做以下内容：

1. 建立 P0 数据表。
2. 导入 50-100 个热门型号标准配件。
3. 实现京东关键词搜索与价格快照入库。
4. 完成简单映射逻辑。
5. 完成基于预算和用途的基础配置生成。
6. 完成 AI 解释接口第一版。

### 25.2 第一个可演示版本应达到的效果

用户输入：

- 预算 5000
- 用途 游戏
- 仅全新

系统输出：

- 一套可用配置
- 每个配件的来源价格
- 总价
- 推荐理由
- 至少 1 套备选方案

### 25.3 当前最关键的工程重点

不是先做漂亮页面，而是先保证：

- 标准型号表可用
- 价格抓取能落库
- 兼容规则能跑通
- 配置结果可追溯
- AI 不乱说

### 25.4 推荐实施顺序

按照 5 模块拆分，建议实施顺序为：

1. 京东数据采集模块
2. 数据与推荐引擎模块
3. AI 模块
4. 应用模块最小版
5. 闲鱼数据采集模块接入

#### 补充说明

- 京东数据采集模块应先于闲鱼接入，因为第一阶段新品价格更稳定，也更适合作为主配置依据。
- 闲鱼数据采集模块建议在主链路跑通后再接入，先作为参考行情层。
- 数据与推荐引擎模块应尽量先做成独立服务，方便后续同时消费京东和闲鱼数据。

1. 数据表
2. 管理后台最小版
3. 京东采集
4. 映射逻辑
5. 配置引擎
6. AI 输出
7. 前台页面
